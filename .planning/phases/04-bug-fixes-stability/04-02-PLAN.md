    **1. 在 import 块末尾添加** `"go.uber.org/zap"`

    **2. 修改 Engine 结构体（第 15 行）**
      - 从 `type Engine struct{}` 改为 `type Engine struct { logger *zap.Logger }`
      - 不修改 `NewEngine()` 的返回——空 Engine 的 logger 为 nil，向后兼容
      - 添加 setter 方法：
        ```go
        func (e *Engine) WithLogger(l *zap.Logger) *Engine {
            e.logger = l
            return e
        }
        ```

    **3. 添加包级 logger 变量（便于全局规则函数访问）**
      - 在 Engine 结构体定义后添加：
        ```go
        var engineLogger *zap.Logger
        ```

    **4. 在 `EvaluateGlobalRules` 中设置包级 logger**
      - 在 `rules := GlobalRules()` 之后添加：
        ```go
        engineLogger = e.logger
        defer func() { engineLogger = nil }()
        ```

    **5. 修复 3 处 json.Marshal 错误处理**
      每处修改遵循同一模式（per D-02, D-03）：
      原代码（3 处位置：第 174、241、303 行）：
      ```go
      evJSON, _ := json.Marshal(evidence)
      ```
      改为带错误检查 + 日志的版本（不包含代码块，以下为详细描述）：
      - `evJSON, marshalErr := json.Marshal(evidence)`
      - 如果 `marshalErr != nil`：
        - 如果 `engineLogger != nil`，记录：`engineLogger.Error("alert engine: failed to marshal evidence", zap.String("rule_id", "<rule_id>"), zap.Strings("evidence_keys", evidenceKeys), zap.Error(marshalErr))`
        - 注意：`evidenceKeys` 是证据 map 的键名列表（用 `slog` 的 or 手动构建），**不记录完整证据内容**（per D-02）
        - 使用 `evJSON = []byte("{}")` 作为空证据 fallback
      - 如果 `marshalErr == nil`，行为不变

      具体要收集的证据键名：
      - evaluateGMVDrop: `evidence` map 的键为 `["current_7d_avg", "prev_14d_avg", "current_value", "baseline_value", "change_rate", "window_7d_count", "window_14d_count", "total_samples", "triggered"]`
      - evaluateLateDeliverySpike: `["current_value", "baseline_value", "change_rate", "window_7d_count", "window_14d_count", "total_samples", "triggered"]`
      - evaluateCancelRateSpike: `["current_7d_avg", "prev_14d_avg", "current_value", "baseline_value", "change_rate", "window_7d_count", "window_14d_count", "total_samples", "triggered"]`

      由于 evidence map 在每处都是字面量构造的，可以直接在错误日志中硬编码键名列表，无需运行时提取。

    **6. 对每个 evaluate 函数，将 evidence 构造提取到 marshal 之前**，以便在日志中引用键名。实际上证据 map 已经构造完毕——只需在 marshal 前准备键名列表字符串切片。

    **7. 保持向后兼容**
      - `NewEngine()` 仍返回 `&Engine{}`（logger 为 nil）
      - 全局规则函数 `evaluateGMVDrop` 等创建 `&Engine{}` 时 logger 为 nil——`engineLogger` 包级变量由 `EvaluateGlobalRules` 注入
      - 当没有设置 logger 时，marshal 错误继续使用空字符串 `evJSON`（行为与修正前相同，但不静默）——但日志会被跳过

    此外，在 `evaluateGMVDrop`（第 138 行）、`evaluateLateDeliverySpike`（第 200 行）、`evaluateCancelRateSpike`（第 267 行）中有一行现有代码 `(&Engine{}).queryMetricSeries(...)`——这些不需要修改，它们只用于查询，不涉及日志。

    **关键约束（per D-03）：**
    - marshal 失败后不跳过告警——`evJSON` 使用空 JSON 对象 `{}` 或空字符串，`Message` 字段仍包含可读信息
    - 告警不能完全丢失

    对 `internal/repository/ontology/repository.go` 做以下修改（per D-07, D-08, D-09）：

    **1. 修复 `fullTableName()` 方法（第 227-229 行）**
      原代码：
      ```go
      func (m tableMapping) fullTableName() string {
          return m.Schema + "." + m.Table
      }
      ```
      改为（使用 pgx.Identifier.Sanitize，参考 compiler.go 中的 V2 模式）：
      ```go
      func (m tableMapping) fullTableName() string {
          // GODEPRECATED: use V2 compiler instead
          return pgx.Identifier{m.Schema, m.Table}.Sanitize()
      }
      ```

    **2. 为 4 个 V1 回退路径添加 allowlist 检查**
      
      在 `QueryByObjectType`、`GetObjectByID`、`GetObjectMetrics`、`SearchObjects` 这 4 个函数的 V1 回退路径中，`objectTableMap[objectType]` 查找后面的代码直接使用 `mapping` 字段构建 SQL。虽然 `objectTableMap` 是编译期常量映射，但添加显式的 allowlist 检查作为防御层。

      具体位置和修改方式：
      
      **A) QueryByObjectType（第 336-338 行）——已存在 mapping 查找**
        现有代码：
        ```go
        mapping, ok := objectTableMap[objectType]
        if !ok {
            return nil, fmt.Errorf("unknown object type: %s", objectType)
        }
        ```
        这已经是一个有效的 allowlist 检查——`objectType` 不在 `objectTableMap` 中时返回错误。但为了记录 `GODEPRECATED` 状态，在 `mapping, ok := ...` 之后添加注释：
        ```go
        // GODEPRECATED: V1 path uses hardcoded objectTableMap; migrate to V2 compiler
        ```

      **B) GetObjectByID（第 437-439 行）——同 QueryByObjectType 模式**
        已有 similar check。添加同上注释。

      **C) GetObjectMetrics（第 497-499 行）——同 QueryByObjectType 模式**
        已有 similar check。添加同上注释。

      **D) SearchObjects（第 573-575 行）——同 QueryByObjectType 模式**
        已有 similar check。添加同上注释。

    **3. 在 `fullTableName()` 返回值被用于 SQL 查询的所有 4 处 V1 回退路径中添加 `// GODEPRECATED: use V2 compiler instead` 注释**
      位置：
      - `QueryByObjectType`：第 346 行 `tableName := mapping.fullTableName()` 上方
      - `GetObjectByID`：第 447 行 `tableName := mapping.fullTableName()` 上方
      - `GetObjectMetrics`：第 510 行 `tableName := mapping.fullTableName()` 上方
      - `SearchObjects`：第 583 行 `tableName := mapping.fullTableName()` 上方

      此外在 `objectTableMap` 变量声明（第 75 行）上方已有 `// GODEPRECATED` 注释——保留不变。

    **4. 在 `objectTableMap` 的使用者上方的注释保持不变**
      不需要修改现有的 GODEPRECATED 注释。

    **关键说明（per D-09）：**
    - V2 compiler（`compiler.go`）已经正确使用 `pgx.Identifier` 消毒——V1 路径现在也遵循相同标准
    - `objectTableMap` 是编译期常量，但防御性消毒是安全最佳实践
    - `sanitizeIdent` 函数（第 793-795 行）已经存在并用于列名——`fullTableName` 的消毒是补全最后一环
