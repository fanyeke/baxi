# v0.5.3 API Gateway Runbook

> **ARCHIVED:** This runbook documents the legacy v0.5.3 Python/FastAPI gateway that has been fully migrated to Go/PostgreSQL. The `python3` commands and scripts referenced below no longer exist. Current API server runs via `make api` (Go chi server on :8080).

## Setup

```bash
pip install -e .
echo "API_BEARER_TOKEN=$(python3 -c \"import secrets; print(secrets.token_urlsafe(32))\")" >> .env
```

## Start

```bash
python3 scripts/run_api.py
# or with custom port:
python3 scripts/run_api.py --port 8765
```

API docs: http://127.0.0.1:8765/docs (requires ENABLE_DOCS=1)
OpenAPI: http://127.0.0.1:8765/openapi.json

## P0 Endpoints

### Health (no auth)
```bash
curl http://127.0.0.1:8765/api/v1/health
# {"status":"ok","version":"0.5.3","db_connected":true}
```

### Status (auth required)
```bash
curl -H "Authorization: Bearer dev-token" http://127.0.0.1:8765/api/v1/status
# Returns table counts + last pipeline run info + migration_status
```

### Alerts (auth required)
```bash
curl -H "Authorization: Bearer dev-token" "http://127.0.0.1:8765/api/v1/alerts?severity=high&limit=10"
```

### Tasks (auth required)
```bash
curl -H "Authorization: Bearer dev-token" "http://127.0.0.1:8765/api/v1/tasks?status=todo&limit=10"
```

### Outbox (auth required)
```bash
curl -H "Authorization: Bearer dev-token" "http://127.0.0.1:8765/api/v1/outbox?status=pending&limit=10"
```

### Dispatch (auth required, dry-run by default)
```bash
# Dry-run (default)
curl -X POST -H "Authorization: Bearer dev-token" -H "Content-Type: application/json" \
  -d '{"channel":"manual","limit":1}' http://127.0.0.1:8765/api/v1/outbox/dispatch

# Apply (real execution)
curl -X POST -H "Authorization: Bearer dev-token" -H "Content-Type: application/json" \
  -d '{"apply":true,"channel":"manual","limit":1}' http://127.0.0.1:8765/api/v1/outbox/dispatch
```

## Auth

All non-/health endpoints require: `Authorization: Bearer <API_BEARER_TOKEN>`

Set in `.env` or env var: `API_BEARER_TOKEN=<generate-with-secrets.token_urlsafe(32)>`

## Error Format

```json
{

  "message": "Authorization header is required",
  "diagnosis": "No Bearer token provided",
  "suggested_action": "Add 'Authorization: Bearer <token>' header"
}
```

## Error Diagnosis (v0.5.3)

When an API call fails, the response includes a `request_id`. Use this to diagnose the failure:

```bash
curl -H "Authorization: Bearer dev-token" \
  "http://127.0.0.1:8765/api/v1/logs/diagnosis?request_id=<request-id>"
```

Response includes:
- `summary`: Human-readable error summary
- `error_code`: Machine-readable error code
- `diagnosis`: Root cause analysis
- `suggested_action`: Recommended fix
- `related_logs`: Array of related error and audit log entries

Example with unknown request_id:
```bash
curl -H "Authorization: Bearer dev-token" \
  "http://127.0.0.1:8765/api/v1/logs/diagnosis?request_id=nonexistent"

# Returns 404: {"error_code":"NOT_FOUND","message":"No logs found for request_id: nonexistent",...}
```

## Migration Health (v0.5.3)

Check database migration status:

```bash
curl -H "Authorization: Bearer dev-token" http://127.0.0.1:8765/api/v1/status | jq .migration_status
```

Example responses:
- `{"status":"ok","failed":[]}` — All migrations applied
- `{"status":"degraded","failed":[{"table":"review_retro","migration":"007_...","error":"..."}]}` — Non-critical failure
- `{"status":"failed","failed":[{"table":"dwd_order_level",...}]}` — Critical table missing

## Troubleshooting

- **API_BEARER_TOKEN not set**: Add to `.env` or export env var
- **DB not found**: API warns on startup, health shows `db_connected: false`
- **Port in use**: Use `--port` flag on `run_api.py`
- **Schema migration**: Applied automatically at startup. Check `/status` for `migration_status`
- **Request failure**: Copy `request_id` from error response, use `/logs/diagnosis?request_id=xxx` to diagnose
