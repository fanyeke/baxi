# 0. GitHub 当前实现确认

## 已完成的关键底座

当前最新修复提交是：

```
b3dab33a fix(ontology-v2): harden ExecuteAction and ApplyService approval gates
```

该提交说明明确移除了 `ExecuteAction` 自动创建 approved proposal 和直接执行，改为 dry-run validation，并强化了 `ApplyService.ExecuteProposal` 的审批门禁。

代码层面也确认：`ExecuteAction` 现在只校验 object action binding、action registry 和 payload，然后返回：

```
{
  "would_execute": true,
  "message": "Use propose_action to create a proposal for execution."
}
```

它不再写 `ai.decision_case`，不再写 `ai.action_proposal`，也不再调用 `ApplyService`。

`ApplyService.ExecuteProposal` 现在也明确要求：

```
proposal.ApplyStatus == "approved"
```

否则返回 `ErrNotApproved`。

# Phase 1：完整 Ontology v2 语义层

```

```

```

region

```

```

```

这三个对 Agent 决策非常有用，尤其是差评、支付异常、物流履约。

------

## 1.1 每个对象的标准模板

每个对象都必须补齐：

```

  source
  properties
  metrics
  relationships
  actions
  alert_fields
  governance
```

| 优先级 | 对象             | 作用                              |
| ------ | ---------------- | --------------------------------- |

| P1     | `region`         | 区域销售和物流异常                |
| P2     | `review`         | 差评、商品质量、卖家服务质量      |

| P2     | `marketing_lead` | 营销转化                          |
| P3     | `global`         | 平台级总览指标                    |

# Phase 3：Pi Agent 只读决策路径

## 目标

让 Pi Agent 能读取 Ontology 和 Context，生成结构化决策，但不写入。

------

只允许这些 MCP 工具：

```

get_object
get_linked_objects
build_context
get_decision_context
list_action_schemas
```

暂时不允许 Agent 自动调用：

```
execute_proposal

```

`execute_action` 当前已经是 validation-only，这一点是安全的。

------

## 3.2 决策流程

```
Pi Agent
  → describe_ontology
  → build_context(case_id, recipe_id)
  → inspect evidence
  → generate structured decision
  → return decision JSON
```

# Phase 4：Pi Agent 受控写入路径

## 目标

让 Pi Agent 可以写入系统，但只能通过受控路径写入：

```
propose_action
  → approved
  → execute_proposal
```

------

## 4.1 允许写入的表 / 路径

允许：

```
ai.llm_decision
ai.action_proposal
audit / lineage logs
ops.outbox_event，必须通过 execute_proposal 间接产生
```

不允许：

```
raw.*
dwd.*
mart.*
gov.*
任意 SQL
```

------

## 4.2 标准写入链路

```
Pi Agent
  → build_context
  → generate decision JSON
  → propose_action
  → ai.action_proposal(status=proposed)
  → approve_proposal
  → execute_proposal(dry_run=true 默认)
  → audit / outbox
```

`propose_action` 当前已经存在，并创建 `proposed` proposal。

`execute_proposal` 当前也只能执行 `approved` proposal。

------

## 4.3 写入安全规则

```
1. Agent 不能裸写数据库
2. Agent 不能直接执行真实动作
3. Agent 只能 propose_action
4. execute_proposal 默认 dry_run=true
5. dry_run=false 需要 env flag + approved proposal
6. 所有 proposal 必须带 evidence_refs
7. 所有 action payload 必须通过 schema validation
8. 所有写入必须可追踪 case_id / recipe_id / context_hash
```

------

## 4.4 需要补的功能

当前 `propose_action` 只写了 action_type、payload、status 等。建议下一步补强：

```
1. action_proposal 增加或填充 evidence_refs
2. action_proposal 增加 context_hash
3. action_proposal 增加 recipe_id
4. ai.llm_decision 写入 Agent 决策结果
5. propose_action 接受 decision_json 或 decision_id
6. approve_proposal 可通过 MCP/CLI 操作
7. execute_proposal 记录 agent trace
```

------

## 4.5 验收标准

```
1. Pi Agent 能创建 proposal
2. proposal 状态为 proposed，不是 approved
3. 未审批 proposal 执行失败
4. 审批后 execute_proposal(dry_run=true) 成功
5. dry_run=false 默认不可用
6. proposal 能追溯 context_hash / evidence_refs / decision_id
7. 没有任何 raw/dwd/mart/gov 写入能力暴露给 Agent
```

## 建议 spec

```
specs/006-pi-agent-controlled-write-path/
```
