# v0.2 DB Backend Design

## Architecture Overview

The v0.2 decision backend introduces a SQLite-backed layer parallel to the existing CSV/JSON file pipeline. It stores facts, states, rules, events, and triggers in a structured local database.

```
CSV Raw Data
    ↓
SQLite (olist_ops.db) ← 12 core tables
    ↓
Ingestion Batches / Data Quality / Metrics
    ↓
Rule Engine (reads alert_rules.yml)
    ↓
Abnormal Events / Strategy Recommendations / Action Tasks
    ↓
event_outbox → Trigger Simulator → GitHub / Feishu CLI / local_cli
```

## 12 Core Tables

| Table | Purpose | Primary Key |
|-------|---------|-------------|
| `pipeline_runs` | Record every execution run | `run_id` |
| `ingestion_batches` | Control what gets imported | `batch_id` |
| `dwd_order_level` | Order-level wide table (19 columns) | `order_id` |
| `dwd_item_level` | Item-level wide table (12 columns) | `item_key` |
| `metric_daily` | Daily KPI metrics (wide table) | `metric_date` |
| `metric_dimension_daily` | Per-dimension metrics | `(date, type, value, name)` |
| `alert_events` | Triggered anomalies from rules | `event_id` |
| `strategy_recommendations` | Heuristic decisions from events | `recommendation_id` |
| `action_tasks` | Execution items for teams | `task_id` |
| `review_retro` | Post-execution analysis | `review_id` |
| `event_outbox` | Pending trigger events for dispatch | `outbox_id` |
| `qoder_jobs` | External execution tracking | `job_id` |

## Data Flow

```
1. db_init.py      → Creates database, 12 tables + indexes
2. db_ingest.py    → CSV → SQLite (full/range/date modes)
3. db_calculate_metrics.py → DWD tables → metric_daily
4. db_rule_engine.py  → metric_daily + alert_rules.yml → alerts/events/tasks
5. db_export_feishu.py → SQLite → 5 Feishu CSVs
6. db_trigger_simulator.py → event_outbox → dispatch simulation
```

## Ingestion Modes

- **full**: Clear and rebuild all DWD tables
- **range**: Import only `purchase_date BETWEEN start AND end`
- **date**: Import only `purchase_date = target_date`

## Rule Engine Design

1. Loads `config/alert_rules.yml`
2. Filters for `dimension=overall` rules (dimension-level deferred)
3. For each rule:
   - Reads metric_daily time series
   - Computes 7-day current window and 14-day baseline window
   - Evaluates condition (e.g., `current_7d_avg < prev_14d_avg * 0.85`)
   - If triggered: writes alert_events, strategy_recommendations, action_tasks, event_outbox
4. Deterministic IDs (`rule_id_date`) ensure idempotency

## Schema Files

- `sql/schema.sql` — 12 CREATE TABLE statements
- `sql/indexes.sql` — Performance indexes for common queries
- `sql/seed_rules.sql` — Initial seeding (minimal)
