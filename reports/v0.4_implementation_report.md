## 一、项目概述

v0.4 的核心目标是将现有的 `event_outbox` 模拟系统（v0.2）升级为 **真实的外部分发系统**。通过适配器模式（Adapter Pattern），实现 `event_outbox → adapter → 外部系统` 的可审计分发链路。

### 架构决策

经过 Oracle + Librarian + Metis 三方审查后，关键决策如下：

| 决策 | 方案 | 原因 |
|------|------|------|
| 飞书集成 | 扩展现有 `FeishuClient`（REST API）| 复用 token 管理、重试、干运行模式 |
| 不再使用 CLI subprocess | 改为继承 FeishuClient | lark-cli 是 Go 二进制，不可 import，subprocess 有开销 |
| 无 dispatch_log 表 | event_outbox 加 4 个字段 + CSV 审计 | 减少 JOIN 复杂度，避免存储膨胀 |
| GitHub adapter | v0.4 仅 dry-run payload 生成 | 聚焦飞书通道，GitHub 留在 v0.4.1 |
| 批量同步 | 继续走 sync_feishu_bitable.py | 已有稳定链路，不重复造轮子 |

## 三、架构详解

### 3.1 数据库变更

```sql
ALTER TABLE event_outbox ADD COLUMN dispatch_attempts INTEGER DEFAULT 0;
ALTER TABLE event_outbox ADD COLUMN last_dispatch_at TEXT;
ALTER TABLE event_outbox ADD COLUMN external_ref TEXT;
ALTER TABLE event_outbox ADD COLUMN adapter_name TEXT;
```

**设计理由**：
- `dispatch_attempts`：最大 3 次，超限标记 `status=failed`
- `last_dispatch_at`：审计时间戳
- `external_ref`：目标系统引用（如飞书 message_id）
- `adapter_name`：追踪哪个 adapter 处理

### 3.2 Adapter 模式

```
ChannelAdapter (ABC)
├── dry_run(event) → dict
└── dispatch(event) → dict

实现类：
├── FeishuAdapter      → 调用 FeishuClient.send_message()
├── GitHubIssueAdapter → 生成 Issue payload（v0.4 dry-run only）
├── LocalCLIAdapter    → 写 CSV 命令建议
└── ManualAdapter      → 标记人工处理
```

每个 adapter 返回统一格式：
```python
{"status": "dispatched|failed|skipped|preview",
 "external_ref": "msg_xxx" | None,
 "error": "error message" | None,
 "message": "human-readable" | None}
```

### 3.3 分发调度器

```bash
# 预览模式（无副作用）
python3 scripts/db_dispatch_outbox.py --dry-run

# 真实分发
python3 scripts/db_dispatch_outbox.py --apply

# 按通道过滤
python3 scripts/db_dispatch_outbox.py --channel feishu --apply

# 限制数量
python3 scripts/db_dispatch_outbox.py --limit 10 --apply
```

状态流转：
```
pending → dispatching（乐观锁）
       → dispatched（成功）
       → failed（失败/超次）
       → pending（重试，attempts < 3）
```

### 3.4 FeishuClient 扩展

```python
def send_message(self, chat_id, content, msg_type="text", dry_run=None):
    """新增方法，复用 _request() + token 管理 + 重试退避 [1,2,4]"""
```

- `dry_run=True`：返回 mock message_id，不调用 API
- `dry_run=None`：沿用构造函数的 `self.dry_run`
- 纯文本消息（v0.4 不支持卡片/富文本）
