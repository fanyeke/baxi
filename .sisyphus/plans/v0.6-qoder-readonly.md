# v0.6 Qoder Read-only API Contract

## TL;DR

> **目标**: 让 Qoder 能安全、只读、可审计地理解 Baxi 系统状态。
>
> **核心原则**: 只验证 Qoder 能否**理解**系统，不验证它能否**执行**系统。
>
> **交付物**:
> - 3 个新 API 端点: `GET /qoder/capabilities`, `GET /qoder/context`, `POST /qoder/reports`
> - 1 个新路由模块: `api/routers/qoder.py`
> - 1 个新服务模块: `services/qoder_service.py`
> - 1 个新 Schema 文件: `api/schemas_qoder.py`
> - 1 个新配置: `config/qoder_capabilities.yml`
> - 1 个新迁移: `sql/migrations/011_qoder_tables.sql`
> - 2 个新数据表: `qoder_runs`, `qoder_reports`
> - 3 个新测试文件
> - 1 份 API 文档更新
>
> **不做**:
> - ❌ 前端 `/console/qoder` 页面
> - ❌ `GET /qoder/reports` 列表查询（如果需要，v0.6.1 再做）
> - ❌ dry-run action、通知适配器、每日增量循环
> - ❌ 修改现有端点行为
>
> **Estimated Effort**: Medium
> **Parallel Execution**: YES - 4 Waves
> **Critical Path**: Migration → Schema/Service → Router → Tests → Final Verification

## Context

### Original Request
在现有 FastAPI 网关基础上，为 Qoder（AI Agent）提供安全的只读 API 契约。Qoder 作为 API 只读消费者，不能执行真实业务动作。

### Key Decisions
1. **包含 POST /qoder/reports**: 只记录 Qoder 自己的巡检报告，不产生业务副作用
2. **新建表，不复用 qoder_jobs**: `qoder_jobs` 语义是外部任务分发，与巡检运行完全不同
3. **只做 API 层**: 前端 `/console/qoder` 页面推到后续版本
4. **以项目实际为主**: 遵循现有 `services/`, `api/schemas.py`, `tests/` 的惯例
5. **TDD**: 先写测试再实现

### Research Findings
- **10 个现有路由**: health, status, alerts, tasks, outbox, logs, feishu, pipeline, diagnosis, governance
- **9 个现有服务**: 在 `services/` 目录，原始 SQL + sqlite3.Row 模式
- **Auth**: Bearer token, `Depends(get_current_user)`, 默认 actor="qoder"
- **qoder_jobs 已存在**: 但语义不符，不扩展
- **测试**: pytest, 33 个文件, 385+ cases, 86% 覆盖率
- **迁移系统**: 10 个迁移文件，启动时自动应用，但**不能创建新表**（只能加列）

### Metis Review Findings
- **ALLOWED_PUBLIC_TABLES** 必须扩展，否则新表对服务层不可见
- **迁移系统需新增表创建路径**（现有 `_migrate_if_needed()` 只加列）
- **需为 /qoder/* 添加独立速率限制类别**（建议 60 req/60s）
- **POST /qoder/reports 需幂等性**（INSERT OR IGNORE）

## Work Objectives

### Core Objective
实现 3 个 Qoder 只读 API 端点，让 Qoder 能安全读取系统状态并记录巡检报告，不触发任何业务动作。

### Concrete Deliverables
- `GET /api/v1/qoder/capabilities` - 返回 Qoder 能力矩阵（read_only 模式）
- `GET /api/v1/qoder/context` - 返回压缩系统上下文（alerts + tasks + outbox + status）
- `POST /api/v1/qoder/reports` - 记录 Qoder 巡检报告（幂等写入）
- `sql/migrations/011_qoder_tables.sql` - 创建 qoder_runs + qoder_reports
- `services/qoder_service.py` - 能力加载、上下文聚合、报告写入
- `api/routers/qoder.py` - 3 个端点路由
- `api/schemas_qoder.py` - Pydantic v2 模型
- `config/qoder_capabilities.yml` - 能力定义配置
- 3 个测试文件
- `docs/API_REFERENCE.md` 更新

### Definition of Done
- [ ] 所有新端点通过 Bearer Token 认证
- [ ] `GET /qoder/capabilities` 返回 read_only 模式 + 允许/禁止端点列表
- [ ] `GET /qoder/context` 不修改任何业务表
- [ ] `POST /qoder/reports` 只写入 qoder_reports，不修改 alerts/tasks/outbox
- [ ] `POST /qoder/reports` 强制验证 `no_apply_performed=true`
- [ ] 所有新端点有 Agent-Executed QA 验证
- [ ] 现有 33 个测试文件 + 385+ cases 不回归
- [ ] 覆盖率保持 86%+

### Must Have
- Bearer Token 认证（复用现有 `get_current_user`）
- read_only 模式明确声明
- 上下文查询不修改业务状态
- 报告写入幂等性（INSERT OR IGNORE）
- 强制 no_apply_performed=true 验证
- 速率限制（独立 qoder 类别）

### Must NOT Have (Guardrails)
- 不允许 Qoder 执行 apply / sync / dispatch
- 不允许直接 SQL / shell / 文件修改
- 不允许修改 alerts / tasks / outbox / config / rules
- 不添加前端页面
- 不添加 dry-run action（v0.6.1）
- 不添加通知适配器（v0.6.2）
