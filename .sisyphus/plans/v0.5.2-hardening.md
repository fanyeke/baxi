## Context

### Original Request
用户审查了 v0.5.1 全面实现报告后，判断当前系统已完成"API Gateway + 前端控制台 Alpha"，但存在 Token 存储不一致、迁移静默失败、Feishu 结果硬编码 0、前端测试缺失等实质性工程问题。下一步应做 **v0.5.2：控制台稳定化与 Qoder 可诊断性增强**，而非直接进入 Qoder 接入。

### Interview Summary
**Key Discussions**:
- P0 优先：Token storage 统一为 sessionStorage / Migration fail-fast / Feishu --json / 前端最小测试
- P1 收敛：alerts/tasks COUNT → service，outbox SQL → dispatch_service，动态表名白名单
- P1 日志：新增 GET /logs/diagnosis?request_id=xxx 聚合错误+审计
- P1 前端：ErrorPanel / EmptyState / LoadingSkeleton / ConfirmApplyDialog 共享组件
- 5 Waves: P0修复 → Service收敛 → 日志诊断 → 前端UX → 验收
- 验收标准明确：12 项可验证条件

### Metis Review
**Identified Gaps** (addressed):
- Feishu 脚本格式：确认为项目内 Python 脚本，需添加 --json flag → 纳入 Task 3
- 审计日志存储：文件 CSV (data/system/api_audit_*.csv) → diagnosis 端点对文件聚合
- Request ID：已在 middleware 生成 (api/main.py:87-105) → 复用无需新建
- 测试范围：最小 12 测试用例 (API client x3 + Dashboard x3 + Alerts x3 + Outbox x3)
- Subprocess 超时：feishu_service.py 已有 timeout=120 → 不需新建，纳入验证

**Guardrails Applied** (from Metis review):
- 不做任何 Schema 变更 (hardening sprint)
- 不引入 vitest + RTL 之外的测试依赖
- 不做 UI 重设计，仅组件提取复用
- 不做完整 Repository 模式，仅消除重复 SQL
- 错误格式变更是 v0.5.2 接受的 breaking change
