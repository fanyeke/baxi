## Context

### Original Request
User proposes wrapping the existing SQLite + event_outbox + adapter framework behind a FastAPI local API service. Qoder will call only the API — never enter project directory, never touch SQLite, never run scripts directly. Core principles: TDD-first, log-first, default dry-run, structured errors with diagnosis/suggested_action.

### Interview Summary
**Key Discussions**:
- Service extraction: Create shared `services/` package used by both CLI (`scripts/db_*.py`) and API (`api/routers/*.py`)
- Schema fix: Include migration to add missing columns (`dispatch_attempts`, `external_ref`, `adapter_name`, etc.) to `event_outbox` and `alert_events`
- Test strategy: ~30 new API tests using httpx TestClient with temp-file SQLite DB
- P1 endpoints: Document as future work only, zero implementation code
- Import strategy: Launcher script `scripts/run_api.py` sets up `PYTHONPATH`, keeping `api/` and `services/` imports clean

**Research Findings**:
- Adapter framework is API-ready: `ChannelAdapter` ABC with `dry_run()` / `dispatch()` methods, registry-based resolution
- `db_dispatch_outbox.py` has clean dispatch logic: `fetch_pending()` → `claim_event()` → `resolve_adapter()` → `write_result()`
- Test infrastructure: 205 existing tests (all sync), 4 test patterns — DB assertions, file checks, adapter unit tests, subprocess integration
- No async/await code anywhere in project — keep everything synchronous
- Schema-code discrepancy: `event_outbox` uses `dispatch_attempts`, `external_ref`, `adapter_name` NOT in `schema.sql`; these are added via migration `005_dispatch_adapters.sql`

### Metis Review
**Identified Gaps** (addressed):
- Import strategy: Use `scripts/run_api.py` launcher with `PYTHONPATH` setup, keep `api/` and `services/` import-clean
- Migration approach: Run existing `005_dispatch_adapters.sql` at API startup if columns are missing (don't rewrite `schema.sql`)
- Edge cases: Empty DB returns `[]` not errors, concurrent dispatch handled by optimistic locking, large responses have default `limit=100`, missing DB returns `db_connected: false`
- Existing tests must pass after service extraction — verify after each extraction step
- Token missing at startup: Fail fast with clear error message
- CORS: Not needed for machine-to-machine local API — explicitly excluded

## Work Objectives

### Core Objective
Build a local FastAPI API service that wraps the SQLite backend and adapter framework, providing controlled read/write endpoints with auth, logging, and structured error responses — all built TDD-first.

- `sql/migrations/006_api_schema_fix.sql` — auto-apply migration for missing columns

- [ ] `python-json-logger` produces valid JSON in `logs/api/api.log` for every request

- 6 P0 endpoints: health, status, alerts, tasks, outbox (GET), outbox/dispatch (POST)
- Bearer Token auth on all non-/health endpoints
- Structured JSON error responses with diagnosis + suggested_action
- 3 JSON log files (api.log, error.log, audit.log) + CSV audit log
- Default dry-run: all write endpoints require explicit `apply: true`
- Schema migration auto-applied at API startup (missing columns added)
- `services/` package shared by CLI and API
- OpenAPI schema auto-generated at `/docs` and exportable to `docs/openapi-v0.5.json`

### Must NOT Have (Guardrails)

- NO raw SQL execution endpoint
- NO raw shell execution endpoint
- NO complex RBAC or multi-user auth
- NO async/await code (keep everything synchronous)
- NO rewriting `sql/schema.sql` (use migration for missing columns)
- NO breaking existing CLI scripts (205 tests must still pass)
- NO new dependencies beyond: fastapi, uvicorn, httpx, python-json-logger
- NO CORS middleware (machine-to-machine local API)
- NO pagination framework (use `?limit=` query param with max 1000)
