## Context

**Upgrade v0.2** (SQLite backend, 12 tables, 74 tests, overall-level alerts) to v0.3 dimensional alerts for seller/category/region.

### Key Decisions
- Exactly 3 dimensions: seller, category, region
- Exactly 6 dimensional rules
- impact_score v1 = severity_weight × sample_size
- alert_id = sha256 with \xff separators for idempotency
- Top N caps: max 50 alerts, 20 recommendations, 40 tasks per run (global)
- event_outbox channels: feishu_cli, github_issue, manual, qoder_pending, local_cli
- No new Feishu tables; enhance existing 5 CSV exports
- New config/dimensional_alert_rules.yml (does NOT merge with alert_rules.yml)

  **What to do**:
  - Create `scripts/db_dimensional_rule_engine.py` — the core dimensional rule engine
  - Supports `--mode full` and `--top-n 50` (default), `--dimension seller|category|region`
  - Reads `metric_dimension_daily`, `config/dimensional_alert_rules.yml`
  - Evaluates each rule against the corresponding dimension's metrics
  - Generates `alert_events` with affected_orders, affected_gmv, impact_score
  - Generates `strategy_recommendations` and `action_tasks` (calls db_generate_recommendations or inline)
  - Writes `event_outbox` with channel routing

  Condition evaluator (support existing 4 patterns + 1 new):
  ```python
  # Pattern 1: "value_gt: X" → current_value > X
  # Pattern 2: "value_lt: X" → current_value < X
  # Pattern 3: "change_rate_lt: X" → change_rate < X (needs baseline)
  # Pattern 4: "change_rate_gt: X" → change_rate > X (needs baseline)
  # Pattern 5 (new): "baseline_minus_current_gt: X" → baseline - current > X
  ```

  Baseline calculation for dimensional metrics:
  - If rule has `baseline_window` (e.g., 14), use avg of previous 14 days
  - Otherwise, use latest value as both current and baseline (no change comparison)

  impact_score formula (simplified v1):
  ```python
  severity_weight = {'high': 3, 'medium': 2, 'low': 1}.get(severity, 1)
  impact_score = severity_weight * sample_size
  ```

  alert_id generation (idempotent):
  ```python
  import hashlib
  raw = f"{rule_id}\xff{metric_date}\xff{dimension_type}\xff{dimension_value}"
  alert_id = f"dim-{hashlib.sha256(raw.encode()).hexdigest()[:12]}"
  ```

  Top N enforcement:
  - After generating all alerts, sort by: severity desc, impact_score desc, sample_size desc
  - Apply global caps from limits section (50 alerts, 20 recommendations, 40 tasks)
  - Mark capped-out alerts as `status='suppressed'` instead of deleting

  event_outbox routing:
  - For each alert, determine target_channel from rule config
  - payload_json includes: event_id, rule_id, target_channel, object_type, object_id, severity, summary, recommended_action, metric_name, current_value, affected_gmv

  Zero-baseline protection:
  - If baseline == 0 and rule requires change_rate, skip the rule for that dimension value (don't divide by zero)
  - Log warning for each skipped dimension

  NULL protection:
  - Skip any rows where dimension_value IS NULL during metric reading
