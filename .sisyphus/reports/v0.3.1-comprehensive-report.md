## 一、背景与目标

### 1.1 问题溯源

v0.3 引入了基于 seller/category/region 三个维度的异常检测引擎（`db_dimensional_rule_engine.py`），能够生成对象级（object-level）的告警事件、策略建议和行动任务。然而，v0.3 交付时存在严重的**展示层断裂**问题：

| 问题 | 严重度 | 影响 |
|------|--------|------|
| 飞书字段映射 key 名与同步脚本不匹配 | **P0 阻断** | `sync_feishu_bitable.py` 读取 `alert_events`/`review_retro`，但 `feishu_field_mapping.yml` 使用了旧名 `metric_alerts`/`execution_reviews`，导致同步静默失败 |
| 飞书 schema 缺 11+ 维度字段 | **P0 阻断** | `affected_orders`、`affected_gmv`、`impact_score` 等关键字段在飞书表中不可见 |
| 策略文本使用内联 f-string | **功能缺陷** | `db_dimensional_rule_engine.py` 直接拼接字符串生成策略描述，而非使用已定义的 `action_templates.yml` 模板——文本质量不可控、不可维护 |
| 任务表缺 target_object 列 | **功能缺陷** | 无法追踪每条任务针对的具体对象（哪个卖家/品类/区域） |
| CSV 列名与 schema field_id 不一致 | **数据断裂** | `db_export_feishu.py` 导出列名使用 DB 原始字段名（如 `event_id`、`owner_role`），但飞书 API 期望 schema 规范的 field_id（如 `alert_id`、`owner`） |

### 1.2 v0.3.1 核心目标

**让 v0.3 生成的对象级事件在飞书中真正可读、可处理、可复盘** — 通过修复展示层、接入模板引擎、创建闭环工具和抽样验证来证明"对象级事件对人有用"。

## 三、关键修复详解

### 3.1 P0：飞书字段映射 key 名称对齐

**问题**：`feishu_field_mapping.yml` 的顶层 key 使用 `metric_alerts`/`execution_reviews`，但 `sync_feishu_bitable.py` 的 `VALID_TABLES` 使用的是 `alert_events`/`review_retro`。这导致同步脚本在读取配置时找不到目标表，**同步静默失败**。

**修复**：
```yaml
# Before (broken)
metric_alerts:
  local_table: data/ads/metric_alerts.csv
  ...
execution_reviews:
  local_table: data/feishu/execution_reviews_for_feishu.csv

# After (fixed)
alert_events:
  local_table: data/feishu/alert_events_for_feishu.csv
  ...
review_retro:
  local_table: data/feishu/review_retro_for_feishu.csv
```

**验证**：
```python
assert set(feishu_field_mapping.keys()) == set(VALID_TABLES)
# PASS: {'daily_metrics', 'alert_events', 'strategy_recommendations', 'action_tasks', 'review_retro'}
```

### 3.2 CSV 列名与 Schema 对齐（column_map 机制）

**问题**：数据库使用 `event_id`、`owner_role`、`metric_date` 等内部列名，但飞书 API 期望 schema 规范化的 `alert_id`、`owner`、`simulated_date`。

**解决方案**：在 `db_export_feishu.py` 的 `EXPORT_SPECS` 中为每个表引入 `column_map`：

```python
'alert_events': {
    'columns': ['alert_id', 'rule_id', ..., 'owner', 'status'],     # schema field_ids
    'column_map': {'alert_id': 'event_id', 'owner': 'owner_role'},  # schema → DB
},
'strategy_recommendations': {
    'columns': ['title', 'detail', 'owner', 'status', ...],                    # schema field_ids
    'column_map': {'title': 'strategy_title', 'detail': 'strategy_detail',
                   'owner': 'owner_role', 'status': 'execution_status'},       # schema → DB
},
```

导出时：CSV header 使用 schema field_ids → 飞书 API 可直接消费。

### 3.3 模板引擎接入

**问题**：`db_dimensional_rule_engine.py` 使用内联 Python f-string 生成策略文本：

```python
# Before (inline, unmaintainable)
f"Rule '{ev['rule_id']}' on {ev['event_date']}. Current: {ev['current_value']}, "
f"Baseline: {ev['baseline_value']}, Change: {ev['change_rate']:.2%}. "
f"GMV: {ev['affected_gmv']}"
```

**修复**：复用 `db_generate_recommendations.py` 中已有的模板引擎：

```python
# After (template-rendered, structured)
import db_generate_recommendations as rec_gen
templates = rec_gen.load_dimensional_templates()
template_name = RULE_TO_TEMPLATE.get(ev['rule_id'])
context = {'object_id': ev['object_id'], 'current_value': ev['current_value'],
           'affected_gmv': ev['affected_gmv'], 'sample_size': ev['sample_size'], ...}
strategy_detail = rec_gen._render_template(
    templates[template_name]['strategy_detail_template'], context)
```

**效果**：模板渲染的策略文本包含完整的业务上下文（问题描述、证据数据、影响范围），且 ALL CAPS 标记关键词（如 `【问题】`、`【证据】`），显著提升可读性。

## 四、新增能力

### 4.1 飞书状态回流闭环（`db_import_feishu_status.py`）

v0.3.1 之前，飞书 → SQLite 的数据流向是单向的（只有导出，没有回流）。人在飞书中修改了任务状态后，本地数据库无法感知。

**新脚本功能**：
- 读取 `pull_feishu_status.py` 产出的快照 CSV
- 按 `_table` 列分离 `action_tasks` 和 `review_retro` 记录
- 仅 UPDATE 匹配记录（不 INSERT 新记录）
- 仅更新 `{status, feedback}` 两列

**HUMAN_FIELDS 保护机制**：
```
local_status ∉ {todo, new, '', None}  →  SKIP（人工已处理）
local_status ∈ {todo, new, '', None}   →  UPDATE（机器状态，允许覆盖）
```

**测试验证**：
- 3 个任务设置为 `in_progress`（模拟人工处理）
- 飞书快照中相同任务状态为 `todo`
- 导入后：3 个任务均保持 `in_progress`，未被覆盖 ✅

### 4.2 飞书看板视图配置（`create_feishu_views.py`）

为日常运维提供 6 个预定义视图，YAML 驱动，支持 dry-run 验证：

| 视图 | 用途 | 过滤条件 |
|------|------|----------|
| `high_impact_alerts` | 高影响告警 | status IN (new, investigating)，按 impact_score 降序 |
| `seller_alerts` | 卖家维度 | object_type = seller |
| `category_alerts` | 品类维度 | object_type = category |
| `region_alerts` | 区域维度 | object_type = region |
| `high_confidence_strategies` | 高置信策略 | confidence IN (high, medium) |
| `object_tasks` | 未完成任务 | status IN (todo, in_progress) |

Dry-run 模式下验证所有引用字段在 schema 中存在，字段不存在时 exit non-zero 并给出明确错误。

### 4.3 通道路由规则配置（`channel_routing_rules.yml`）

从 `dimensional_alert_rules.yml` 中提取并集中管理 6 条规则的路由配置：

| 规则 | 严重度 | 通道 |
|------|--------|------|
| seller_late_delivery_spike | high | feishu_cli |
| seller_review_score_drop | medium | local_cli |
| category_gmv_drop | medium | feishu_cli |
| category_low_review_cluster | medium | local_cli |
| region_cancel_rate_spike | medium | manual |
| region_late_delivery_spike | high | feishu_cli |

## 八、遗留问题与建议

### 8.1 Schema 命名不一致

- `due_at`（DB 列名）vs `deadline`（schema field_id）
- `execution_status`（DB 列名）vs `status`（schema field_id）
- **建议**：在 v0.4 中统一命名规范，避免 column_map 的维护成本

### 8.2 非维度策略仍使用内联字符串

`db_generate_recommendations.py` 中的 `heuristic_strategy` 来源任务仍使用内联 f-string。
- **建议**：如果 v0.4 需要扩展模板覆盖范围，可将所有策略类型统一接入模板引擎。

### 8.3 create_feishu_views.py 的 --apply 模式为 stub

当前 `--apply` 仅打印 "API call would be made"，未实际调用 Feishu API。
- **建议**：当 Feishu 环境就绪后，对接 `feishu_client.py` 的 `FeishuClient` 完成真实视图创建。

### 8.4 飞书同步链路尚未端到端验证

由于 `--dry-run` 约束，`sync_feishu_bitable.py` 和 `pull_feishu_status.py` 的真实飞书 API 调用尚未在 v0.3.1 中验证。
- **建议**：在 Feishu 沙箱环境中运行 `--apply` 模式，验证完整的 同步→人工修改→回流 闭环。
